# SkyAnchor 最终演示脚本

## 1. 演示目标

本轮最终演示主线固定为：

`用户下单 -> 配送端选择无人机 0/1 -> 开始配送 -> ESP32 收到目标号 -> 订单状态推进`

本版不再使用：

- 用户手动填写目标点位
- 配送端自动分配默认无人机
- 中间下单页

## 2. 固定演示参数

- 用户 ID：`receiver_003`
- 自动目标号：`3`
- 可调度无人机：`0`、`1`
- 小程序后端地址：`http://127.0.0.1:8000`

说明：

- 当前系统会根据用户 ID 末尾数字自动生成目标号。
- `receiver_003` 会对应 `target_id = 3`。
- 配送端必须手动选择 `0` 或 `1`，然后才能开始配送。

## 3. 演示前检查

### 后端检查

1. 打开 `skyanchor-server` 目录。
2. 运行：

```powershell
.\run.ps1
```

3. 浏览器打开：

```text
http://127.0.0.1:8000/health
```

4. 确认返回里至少有这些内容：

```json
{
  "ok": true,
  "mqtt_started": true,
  "default_device": "0",
  "device_candidates": ["0", "1"]
}
```

### 板端检查

串口日志至少出现：

- `wifi got ip`
- `EMQX mqtt connected`
- `system ready`

### 小程序检查

1. 微信开发者工具重新编译。
2. 进入首页。
3. 先走一次 `我是用户`。
4. 再切到 `我是配送员`。

## 4. 正式演示顺序

### 第一步：用户下单

1. 首页点击 `我是用户`。
2. 用户页确认 `用户 ID = receiver_003`。
3. 点击 `保存信息`。
4. 点击 `我要下单`。
5. 弹出确认框后，点击 `确定`。

预期结果：

- 成功弹出“订单已提交”。
- 自动进入订单详情页。
- 详情页显示：
  - 用户 ID：`receiver_003`
  - 目标号：`3`
  - 设备：`待分配`
  - 状态：`待处理`

### 第二步：配送端分配无人机

1. 返回首页点击 `我是配送员`。
2. 在 `待处理订单` 中打开刚创建的订单。
3. 点击 `选择无人机`。
4. 在弹出的列表里选择 `0` 或 `1`。

预期结果：

- 页面提示“已分配 0”或“已分配 1”。
- 订单详情里的设备字段变成所选无人机。
- 主按钮从 `选择无人机` 变成 `开始配送`。

### 第三步：开始配送

1. 在同一订单详情页点击 `开始配送`。

预期结果：

- 页面提示“已开始配送”。
- 后端向所选无人机发布 `start_task`。
- `start_task` 中会带上：
  - `order_id`
  - `target_id = 3`
  - `request_id`

### 第四步：观察状态推进

配送端订单详情页会自动轮询。

预期看到状态逐步推进为：

- `待启动`
- `配送中`
- `标签匹配`
- `执行中`
- `已送达`

用户端再次进入该订单，也能看到同一笔订单的状态变化。

## 5. 现场讲解建议

建议按下面这套话术讲，最稳：

1. “用户端这里只负责下单，不再选择目标点位。”
2. “系统根据当前用户身份自动生成目标号，这里是 3 号目标。”
3. “配送端作为调度角色，不自动分配设备，而是手动选择无人机 0 或 1。”
4. “开始配送后，后端会把订单号和目标号一起下发给 ESP32。”
5. “板端识别目标号后回传状态，小程序实时更新订单状态。”

## 6. 异常场景处理

### 场景 A：点击开始配送没反应

先检查：

- `/health` 是否 `mqtt_started=true`
- 板端日志是否已经连接 MQTT

### 场景 B：用户下单时报网络错误

先检查：

- `skyanchor-server` 是否还在运行
- 微信开发者工具是否已经重新编译
- `/health` 是否能正常打开

### 场景 C：配送端没有看到 `0` 和 `1`

先检查 `skyanchor-server/.env`：

```text
MQTT_DEFAULT_DEVICE=0
MQTT_DISPATCH_DEVICES=0,1
```

改完后要重启后端。

### 场景 D：板端暂时没推进状态

先不要来回乱点按钮。

优先做：

1. 在订单详情页点 `刷新`
2. 看串口日志有没有收到 `start_task`
3. 确认 `target_id=3` 是否已经送到板端

## 7. 最稳的演示节奏

如果答辩现场时间紧，建议只演下面这一条：

1. 启动后端
2. 展示 `/health`
3. 用户端下单
4. 配送端选 `0`
5. 点击 `开始配送`
6. 展示订单状态变化

如果时间更充足，再补：

1. 重新建一单
2. 配送端改选 `1`
3. 说明系统支持多无人机手动调度

## 8. 演示结束判定

本轮最终版的成功标准是：

- 用户可以直接确认下单
- 配送端只能手动选择 `0` 或 `1`
- 未分配时不能开始配送
- 开始配送后 ESP32 能收到目标号
- 订单状态能从创建态推进到后续状态
